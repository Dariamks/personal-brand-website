## 项目背景与要解决的问题

朋友在做猎头业务，每天人工筛简历占工作时间的 60%。具体痛点：

1. **简历格式混乱**——PDF / Word / 图片简历都要先转成结构化数据
2. **JD 匹配靠手感**——HR 凭印象判断"匹配度"，不可解释、不可比较
3. **重复劳动**——同一个候选人换不同包装投到不同公司，HR 要重新读一遍

我决定独立做一个 AI HR 助手，把简历筛选这件事自动化到只剩"最后人工 review 一道"。这也是我把后端经验（FastAPI/PostgreSQL/Celery）跟 LLM 应用（LangChain/Prompt 工程/RAG）结合的一次实战练手。

## 技术架构与选型理由

- **解析层**：先 pdfplumber 提取文本，再正则抽取明显字段（电话、邮箱），最后 LLM 抽取自由格式字段（工作经历、技能）。**多层 fallback 比单纯 LLM 抽取的准确率高 8 个点。**
- **匹配层**：TF-IDF（关键词覆盖）+ 词向量（语义相似度）双通道融合，比单用任一种都准。
- **异步层**：Celery + Redis 队列，把"上传 → 解析 → 评分"做成异步任务，HR 不用看转圈。
- **去重层**：对简历正文 SimHash 指纹化，已见过的简历直接复用解析结果。
- **数据层**：PostgreSQL 存解析结果 + 评分历史，方便后续做模型微调的数据回流。

选 Python 而不是 Java 主要因为 LangChain 生态在 Python 上最完整；选 FastAPI 因为它的 Pydantic schema 校验在 LLM 输出验证场景特别合适。

## 我的角色与个人贡献

**完全独立开发**：

- **简历解析引擎**：设计三层 fallback 抽取策略（pdfplumber → 正则 → LLM）；引入置信度评估，低置信字段标红供人工修正，**修正数据回流模型微调**形成闭环。**解析准确率 96%**。
- **智能匹配算法**：实现 TF-IDF + 词向量双通道融合；构建四维评分模型（技能 40% + 经验 30% + 学历 20% + 稳定性 10%）；提供分数拆解视图让 HR 知道"为什么这个候选人 85 分"。
- **异步解析队列**：Celery + Redis Pub/Sub，把单份简历**串行 60s 优化到并行 12s/份**（5× 提升）。
- **简历指纹去重**：SimHash 算法对正文指纹化，已见过的简历命中率约 30%，**节省 LLM API 调用成本约 30%**。
- **高亮反查**：在最终结果中保留每个评分依据在原文中的位置，HR 一键定位简历原文证据。
