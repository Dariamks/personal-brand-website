## 项目背景与要解决的问题

国家电网科研管理体系下有上万名科技人员，传统的人才评估依赖人工填表与离线打分，存在三个问题：

1. **数据维度割裂**——基础信息、科研成果、项目经历、专利论文、技术方向、岗位能力分散在不同系统，缺少统一画像
2. **评分规则僵化**——业务方想调"成果 vs 项目 vs 能力"权重，要走需求 → 开发 → 上线流程，迭代周期以周计
3. **批量评分慢**——对一个 500 人团队按多级指标（一级 → 二级 → 三级）评分，最初版本需要约 **12 秒**，决策会议上等待感明显

平台目标是把多源数据收口为人才画像，配上**业务方可自助配置规则**的评分引擎，并能秒级出结果。

## 技术架构与选型理由

- **存储层**：MySQL 存画像主表、标签表、规则表、评分结果表；Redis 缓存中间分值与重复子树
- **评分引擎**：纯 Java 实现，不引外部规则引擎（如 Drools），保持复杂度可控、调试友好
- **接口层**：Spring Boot + MyBatis-Plus + Swagger，前端 Vue.js + Element UI

选型的核心考量是 **递归 + 缓存可以解决树形评分的性能问题**，没必要上重型规则引擎。

## 我的角色与个人贡献

实习期间作为后端开发参与，重点贡献：

- **画像数据建模**：参与设计 6 大维度、30+ 结构化标签（基础信息 / 科研成果 / 项目经历 / 专利论文 / 技术方向 / 岗位能力）；提供按标签、按能力、按方向的多条件组合检索接口。
- **可配置评分引擎（核心）**：设计基于权重配置表的评分模型，业务方在后台调整权重（成果 40% + 项目 30% + 能力 20% + 匹配 10%）即时生效；评分结果附带明细拆解，让"为什么这个人是 87 分"对业务方完全可解释。
- **递归算法（核心难点）**：组织架构（部门 → 团队 → 人员）与指标体系（一级 → 二级 → 三级指标）都是树形结构。用递归实现：
  - **DFS 拼装**父子节点形成完整树
  - **子节点分值按权重向上汇总**到父节点
  - **父级指标回溯计算 + 剪枝**，跳过已经达标的子树
- **性能优化**：用 Redis 缓存中间评分结果 + 函数级 Memoization 消除重复子树计算，**500+ 人员批量评分从 ~12s 降至 3s 以内**（4 倍提升）。
- **接口与前后端联调**：负责画像查询、评分计算、评分明细、人员对比等 20+ 接口；参与 SQL EXPLAIN 慢查询分析与索引优化。
