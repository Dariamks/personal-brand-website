## 项目背景与要解决的问题

矿区现场分布着上千台 PLC、传感器与视频终端，每天产生 **200W+ 条**温度、压力、瓦斯浓度等实时数据。原有的监控方式分散在各厂商私有平台中，存在三个核心痛点：

1. **多协议接入混乱**——TCP、MQTT、HTTP RESTful 三种接入方式没有统一抽象，每接入一个新设备都要改业务代码
2. **告警信号噪声大**——一条故障在不同维度被多次触发，告警泛滥导致一线运维"狼来了"现象严重
3. **大屏首屏卡顿**——按区域、设备类型、时间维度做多维聚合的 SQL 查询动辄 3 秒以上，指挥中心大屏体验差

平台目标是把所有协议、设备、告警与可视化收口到一个后台统一治理。

## 技术架构与选型理由

整体采用 **Spring Cloud 微服务 + Netty 自建 TCP 网关** 的混合架构：

- **接入层**：用 Netty 自建 TCP 长连接服务对接 PLC（私有报文协议），用 EMQX + MQTT 标准协议接入新一代传感器，用 Spring WebFlux 拉取第三方系统的 RESTful 数据。三种接入统一抽象成"设备数据事件"投递到内部总线。
- **告警层**：Redis Stream 作为告警总线，下游消费者各自订阅做分级、去重、聚合；告警结果通过 WebSocket 主动推送到大屏。选 Redis Stream 而不是 RocketMQ 的原因——告警链路对延迟敏感（要求秒级到屏），而 Redis Stream 单机吞吐已能覆盖 200W/天的体量，运维复杂度更低。
- **存储层**：MySQL 8.0 + 按月分表存告警历史；Redis 缓存字典与热点指标。
- **可视化**：Vue 3 + ECharts 大屏，全屏 60 秒自动刷新 + WebSocket 增量推送。

## 我的角色与个人贡献

作为后端主力开发，独立或主导完成：

- **多协议数据接入网关**：基于 Netty 实现 TCP 接入服务，处理报文解析、字段映射、心跳保活、断线重连、异常过滤。对接 EMQX 实现 MQTT 主题订阅与下行命令。**接入可用率 99.9%**。
- **高级报警系统（独立设计）**：定义"健康等级、报警泛滥率、10min 峰值报警、确认及时率"四个核心 KPI；用 Redis Stream + Lua 脚本实现告警分级（一般/重要/紧急）、同源告警去重与告警风暴抑制；通过 WebSocket 推送大屏，告警从触发到上屏 **< 1 秒**。
- **可视化大屏指标接口**：负责全厂报警分布、每日/每月报警数、各车间 KPI、自控率与联锁投用率等聚合接口。结合 Redis 二级缓存，将大屏首屏加载从 **3s 降至 600ms 以内**。
- **告警历史查询性能优化**：分析慢查询定位到无索引的范围扫描，落地组合索引 + SQL 重写 + 按月分表方案，查询响应从 **2s+ 降至 200ms 左右**；同时引入字典与配置数据的 Redis 缓存，**数据库 QPS 压力下降约 40%**。
- **核心业务模块**：设备管理、人员/车辆管理、巡检任务、字典配置等模块的接口设计与实现，搭配统一响应结构、参数校验与全局异常处理保障接口规范性。
